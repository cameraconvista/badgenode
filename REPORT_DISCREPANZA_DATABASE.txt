
# REPORT DISCREPANZE DATABASE SUPABASE - BADGEBOX
========================================================

## PROBLEMA IDENTIFICATO
Dal backup CSV allegato emerge una discrepanza critica tra la struttura del database attuale e quella utilizzata dal frontend.

## STRUTTURA ATTUALE DEL DATABASE (dal backup CSV)

### Tabella: public.utenti
```sql
CREATE TABLE public.utenti (
  id uuid NOT NULL DEFAULT gen_random_uuid(),
  nome text,
  cognome text,
  email text,
  pin text,
  ruolo text DEFAULT 'dipendente'::text,
  created_at timestamp without time zone DEFAULT now()
);
```

### Tabella: public.timbrature  
```sql
CREATE TABLE public.timbrature (
  id uuid NOT NULL DEFAULT gen_random_uuid(),
  user_id uuid,
  tipo text,
  data date NOT NULL,
  ora time without time zone NOT NULL,
  giorno_logico date,
  created_at timestamp without time zone DEFAULT now()
);
```

### Foreign Key presente:
```sql
ALTER TABLE public.timbrature 
ADD CONSTRAINT timbrature_user_id_fkey 
FOREIGN KEY (user_id) REFERENCES public.utenti (id);
```

## STRUTTURA USATA DAL FRONTEND

Il codice JavaScript utilizza:
- `pin` (INTEGER) per identificare gli utenti
- Campi denormalizzati `nome`, `cognome` nella tabella timbrature
- Nessun uso di `user_id` UUID

## DISCREPANZE CRITICHE

1. **Identificazione utenti**: 
   - DB: usa UUID (`utenti.id`) + PIN testuale (`utenti.pin`)
   - Frontend: usa PIN numerico come chiave primaria

2. **Relazioni**: 
   - DB: `timbrature.user_id` → `utenti.id` (UUID)
   - Frontend: `timbrature.pin` → `utenti.pin` (INTEGER)

3. **Campi mancanti nel DB**:
   - `timbrature.pin` (INTEGER)
   - `timbrature.nome` (TEXT) 
   - `timbrature.cognome` (TEXT)

4. **Trigger presente ma non utilizzato**:
   - `trg_giorno_logico` calcola automaticamente `giorno_logico`
   - Frontend gestisce questo calcolo manualmente

## SOLUZIONI POSSIBILI

### Opzione A: Aggiornare il Database (CONSIGLIATA)
Aggiungere i campi mancanti mantenendo la struttura UUID:

```sql
-- Aggiungi campi necessari al frontend
ALTER TABLE public.timbrature 
ADD COLUMN pin INTEGER,
ADD COLUMN nome TEXT,
ADD COLUMN cognome TEXT;

-- Crea indice per performance
CREATE INDEX idx_timbrature_pin ON public.timbrature(pin);

-- Aggiorna constraint PIN unico come INTEGER
ALTER TABLE public.utenti 
ALTER COLUMN pin TYPE INTEGER USING pin::INTEGER;
```

### Opzione B: Aggiornare il Frontend
Modificare tutto il codice per usare UUID invece di PIN.

## RACCOMANDAZIONI IMMEDIATE

2. Decidere se mantenere PIN numerici o passare a UUID
3. Aggiornare le tabelle o il codice di conseguenza

- **Criticità**: ALTA - l'app non può funzionare con questa discrepanza
- **Tempo stimato**: 2-4 ore per la migrazione completa
